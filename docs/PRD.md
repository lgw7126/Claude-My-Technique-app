# PRD — Technique Studio (가칭)

영상을 장면 단위로 자동 분해하고 연출 기법을 분석해 나만의 레퍼런스 라이브러리로 축적하는 로컬 웹앱.

- **버전**: v0.1 (초안)
- **작성일**: 2026-07-15
- **구조**: 공통 코어 1개 + 주제 프리셋 2개 (Viral Grammar, Runway Studio)
- **파일럿 주제**: Viral Grammar (트렌딩 쇼츠 분석)

---

## 1. 배경과 목표

### 1.1 배경

영상 레퍼런스 스터디의 실제 작업 흐름은 "수십 개를 찾고 → 비교하고 → 다시 정리"의 반복이다.
이 과정을 하나의 시스템으로 만든다: 영상이 들어오면 장면이 자동으로 나뉘고, 각 장면의
기법(색·리듬·구성)이 분석되어, 검색 가능한 지식 라이브러리로 쌓인다.

### 1.2 목표

| # | 목표 | 측정 기준 |
|---|------|-----------|
| G1 | 손대지 않는 전자동 파이프라인 | 앱을 열면 최신 트렌딩 쇼츠의 분석 리포트가 이미 준비되어 있다 |
| G2 | API 의존 최소화 | 핵심 분석(장면·색·리듬)은 100% 로컬 연산. 외부 API는 트렌딩 목록 수집(YouTube Data API)만 |
| G3 | 주제 확장 가능한 코어 | 분석 스키마 파일 교체만으로 새 주제(런웨이 등) 추가 |
| G4 | 개인용→공개용 전환 대비 | 영상 확보 모듈을 스위치 방식으로 분리 (auto ↔ upload) |

### 1.3 비목표 (Non-Goals)

- 다중 사용자 / 로그인 / 클라우드 저장 (v1은 개인용 로컬 앱)
- 영상 편집 기능 (분석 전용)
- TikTok / Instagram Reels 수집 (v1은 YouTube Shorts만; 구조는 확장 가능하게)
- AI 해설 자동 생성 (v2에서 Claude API 옵션으로; v1은 자리만 마련)

## 2. 사용자와 핵심 시나리오

**주 사용자**: 비주얼 아티스트 / 영상 크리에이터 1인 (앱 소유자 본인)

### 시나리오 A — 아침 브리핑 (전자동)
1. 밤사이 스케줄러가 트렌딩 쇼츠 목록을 수집하고, 영상을 받아 분석 후 원본을 삭제한다.
2. 아침에 앱을 열면 좌측 라이브러리에 신규 분석 리포트 N건이 "NEW" 배지와 함께 떠 있다.
3. 훅 유형·컷 빈도 순으로 정렬해 훑고, 흥미로운 것만 상세 화면에서 장면 단위로 확인한다.

### 시나리오 B — 정밀 해부 (선택적)
1. 특정 영상의 장면 타임라인에서 씬 03을 클릭한다.
2. 우측 패널에서 해당 장면의 컷 리듬, 컬러 팔레트, 자막 타이밍을 확인하고 메모/태그를 단다.
3. 태그는 라이브러리 전체 검색에 반영된다 ("첫 1초 얼굴 클로즈업 훅"만 모아보기).

### 시나리오 C — 주제 전환 (프리셋)
1. 상단 주제 스위처에서 "Runway"를 선택한다.
2. 같은 UI에 런웨이 분석 스키마(실루엣/소재/조명)가 로드된다. 라이브러리는 주제별로 분리 보관.

## 3. 시스템 구성

```
┌────────────────────────────────────────────────────────┐
│                     Frontend (Next.js)                  │
│  좌: 라이브러리   중: 플레이어+장면 타임라인   우: 분석 패널  │
└───────────────────────────┬────────────────────────────┘
                            │ REST (localhost)
┌───────────────────────────┴────────────────────────────┐
│                   Backend (FastAPI, Python)             │
│                                                          │
│  [수집기 Collector]     YouTube Data API → 트렌딩 목록    │
│  [확보기 Fetcher]★     mode=auto: yt-dlp 자동 다운로드   │
│                        mode=upload: 파일 업로드 대기     │
│  [분석기 Analyzer]     ffmpeg + PySceneDetect (전부 로컬) │
│    - 장면 분할 (컷 감지)                                  │
│    - 편집 리듬 (컷 빈도, 평균 샷 길이, 첫 1초 구성)         │
│    - 컬러 팔레트 (장면별 k-means)                          │
│    - 오디오 에너지 곡선 (사운드 싱크 감지)                  │
│  [정리기 Janitor]      분석 완료 → 원본 영상 삭제,         │
│                        썸네일+분석 데이터만 보관           │
│  [스케줄러]            매일 지정 시각 자동 실행             │
└───────────────────────────┬────────────────────────────┘
                            │
                     SQLite + 로컬 파일 (썸네일/스프라이트)
```

★ **Fetcher 모드 스위치**: `config.yaml`의 `fetch_mode: auto | upload` 한 줄로 전환.
auto는 개인 연구용(YouTube 약관 위반 소지 인지하고 사용), upload는 공개 배포용.
Fetcher는 독립 모듈로 격리하여 어느 쪽이든 Analyzer 이후 파이프라인은 동일.

### 3.1 데이터 모델 (요약)

- **Video**: id, 주제(preset), 제목, 채널, 조회수, 좋아요, 게시시각, 수집시각, 길이, 상태
- **Scene**: video_id, 순번, 시작/끝(ms), 썸네일, 컬러팔레트(hex[]), 컷수, 평균샷길이
- **Analysis**: video_id, 훅타입(자동추정+수동수정), 컷빈도(cuts/10s), 첫1초구성, 오디오싱크점수
- **Tag / Note**: scene_id 또는 video_id에 사용자가 부착. 전역 검색 대상
- **Preset**: 주제별 분석 스키마(카테고리, 라벨, 태그 후보) — JSON 파일

### 3.2 분석 항목 정의 (Viral Grammar 프리셋)

| 항목 | 산출 방법 | 자동화 |
|------|-----------|--------|
| 장면 분할 | PySceneDetect 콘텐츠 감지 | 전자동 |
| 컷 빈도 | 장면 수 ÷ 길이 (cuts/10s) | 전자동 |
| 평균 샷 길이 | 장면 길이 평균/중앙값 | 전자동 |
| 첫 1초 구성 | 0~1s 프레임 추출 + 밝기/움직임/얼굴존재(로컬 검출) | 전자동 |
| 컬러 팔레트 | 장면별 대표 프레임 k-means 5색 | 전자동 |
| 오디오 싱크 | 오디오 에너지 피크와 컷 지점 일치율 | 전자동 |
| 자막 타이밍 | 프레임 하단부 텍스트 영역 변화 감지 | 반자동(v1은 감지만) |
| 훅 유형 분류 | 위 지표 기반 규칙 분류 + 수동 수정 | 반자동 |

### 3.3 Runway Studio 프리셋 (2차 적용)

동일 코어 위에 스키마만 교체: 장면 분할·컬러 팔레트·편집 리듬은 그대로 재사용.
런웨이 특화 항목(실루엣, 소재 질감, 조명 방향, 워킹 앵글)은 태그 체계 + 수동 라벨링 중심,
컬러 팔레트는 "시즌 컬러 보드" 뷰로 재구성. 수집기는 지정 채널(패션위크 공식 채널) 구독 방식.

## 4. 화면 설계

스크린샷 레퍼런스(MV Technique Studio)의 3단 레이아웃 계승, 다크 테마.

1. **좌측 — 라이브러리**: 수집일 기준 목록. 썸네일, 제목, 조회수, 분석 상태 배지.
   정렬(조회수/컷빈도/수집일), 필터(훅유형/태그), 검색.
2. **중앙 — 뷰어**: 상단 임베드 플레이어(YouTube iframe) + 하단 장면 타임라인
   (장면 썸네일 스트립, 클릭 시 해당 구간 재생·우측 패널 갱신).
3. **우측 — 분석 패널**: 선택 장면의 분석 카드 스택 — 편집 리듬, 컬러 팔레트,
   첫 1초 구성, 오디오 싱크, 태그/메모 입력. 하단에 "AI 해설" 버튼 자리(v2, 비활성).
4. **상단 바**: 주제 프리셋 스위처, 수동 추가(URL/파일), 설정(수집 시각, fetch_mode, API 키).

## 6. 리스크와 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 자동 다운로드의 YouTube 약관 위반 소지 | 공개 배포 시 법적/계정 리스크 | Fetcher 모듈 격리 + fetch_mode 스위치. 원본 즉시 삭제, 분석 데이터만 보관. 공개 시 upload 모드 강제 |
| YouTube Data API 쿼터/키 필요 | 수집기 동작 불가 | 무료 쿼터(일 10,000 유닛)로 충분. 키 미설정 시 수집기만 비활성, 나머지 기능 정상 |
| yt-dlp 잦은 업데이트 필요 | 다운로드 실패 | Fetcher에 실패 재시도 + 버전 자가 점검. 실패 영상은 "메타데이터만" 상태로 보관 |
| 개발 환경(현 원격 세션)의 네트워크 정책이 pypi/npm 차단 | 의존성 설치 불가 → 빌드 불가 | **빌드 시작 전 환경의 네트워크 정책 완화 필요** (또는 로컬 환경에서 빌드) |
| 쇼츠 세로 영상과 3단 레이아웃 충돌 | 중앙 뷰어 여백 | 세로 플레이어 + 좌우 여백에 핵심 지표 오버레이 배치 |
